
一、总体架构规范

1.1 分层架构设计原则

架构分层说明
系统采用标准的六层架构设计，各层职责明确，确保代码结构清晰、易于维护和扩展：

Controller层 → Service层 → Dao层 → Entity层
     ↓              ↓           ↓          ↓
   Dto层         Business      Mapper    Database
     ↓              ↓
   Util层        Config层
     ↓
Exception层 → Common层

各层职责定义
Controller层：负责HTTP请求接收、参数校验、业务调用、响应封装
Service层：实现核心业务逻辑，处理事务管理，调用Dao层进行数据操作
Dao层（Mapper层）：负责数据库CRUD操作，使用MyBatis-Plus简化数据访问
Entity层：实体类映射数据库表结构，包含基本字段和关联关系
Dto层：数据传输对象，用于前后端数据交互，避免暴露内部实体细节
Util层：通用工具类封装，包括日期处理、加密解密、文件操作等
Config层：系统配置类，包括数据库、Redis、AI API、安全等配置
Exception层：全局异常处理，统一异常响应格式
Common层：通用常量、枚举、统一响应结果封装

依赖注入规则
各层之间通过Spring依赖注入进行协作
禁止跨层直接调用（如Controller直接调用Dao层）
Service层可调用其他Service，但需谨慎避免循环依赖

普通变量：小驼峰命名，如workOrderId、repairStatus

2.2 代码格式规范

缩进与空格
使用4个空格进行缩进，禁止使用Tab键
运算符两侧添加空格：int result = a + b;
逗号后添加空格：method(param1, param2, param3)
if/for/while等关键字后添加空格：if (condition) {

行长度限制
单行代码不超过120个字符
超长行使用换行，运算符放在行首：
String sql = "SELECT * FROM work_order " 
           "WHERE status = ? " 
           "AND create_time > ?";

大括号规范
左大括号不换行，右大括号单独一行：
if (condition) {
    // code
}

注释规范
类注释：说明类的用途、作者、创建时间
方法注释：使用Javadoc格式，说明功能、参数、返回值、异常
复杂逻辑注释：在代码上方添加行内注释，解释业务逻辑
TODO注释：标记待完成的功能，格式：// TODO: 说明

四、数据库规范

4.1 表设计规范

命名规范
表名：小写+下划线，复数形式，如work_orders、asset_categories
字段名：小写+下划线，如work_order_id、fault_type
索引名：idx_{表名}_{字段名}，如idx_work_orders_status
外键名：fk_{子表名}_{父表名}，如fk_work_orders_users

字段类型选择
业务场景   推荐类型   说明
主键ID   BIGINT   自增主键

状态字段   TINYINT   1-127范围足够

金额字段   DECIMAL(10,2)   精确计算

文本字段   VARCHAR(255)   一般文本

长文本   TEXT   超过255字符

日期时间   DATETIME   精确到秒

JSON数据   JSON   MySQL 5.7+

必填字段
每个表必须包含以下字段：
id：主键，BIGINT类型
create_time：创建时间，DATETIME类型
update_time：更新时间，DATETIME类型
is_deleted：逻辑删除标志，TINYINT类型（0-未删除，1-已删除）

5.2 全局异常处理器

统一异常处理
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity> handleBusinessException(BusinessException e) {
        ApiResponse response = ApiResponse.error(e.getErrorCode(), e.getMessage());
        return ResponseEntity.badRequest().body(response);
    }
    
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldError().getDefaultMessage();
        ApiResponse response = ApiResponse.error("VALIDATION_ERROR", message);
        return ResponseEntity.badRequest().body(response);
    }
    
    @ExceptionHandler(Exception.class)
    public ResponseEntity> handleGenericException(Exception e) {
        log.error("Unexpected error", e);
        ApiResponse response = ApiResponse.error("INTERNAL_ERROR", "系统内部错误");
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(response);
    }
}

六、安全规范

6.1 认证授权规范

JWT Token机制
Token结构：Header.Payload.Signature
Payload内容：用户ID、角色、过期时间
签名算法：HS256
有效期：Access Token 2小时，Refresh Token 7天

权限控制
基于角色的访问控制（RBAC）
方法级权限：使用@PreAuthorize注解
数据权限：在Service层实现部门级、个人级数据过滤

@PreAuthorize("hasRole('ADMIN') or hasPermission(#workOrderId, 'work_order:read')")

6.2 数据安全规范

敏感数据处理
密码存储：使用BCrypt加密
手机号显示：中间4位脱敏（138****1234）

日志记录：敏感信息不在日志中记录
